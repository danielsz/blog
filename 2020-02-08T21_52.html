Next-gen Clojure tooling

When we we say that Clojure is hosted on the JVM, we often forget the corollary, that Clojure tooling is built on Maven. We’d be forgiven for the oversight: the tooling is good at keeping Maven out of sight. But Maven has been lurking in the dark all along. It is everywhere: behind Clojars with the Maven repository format, behind Boot with pomegranate as the “sane Clojure API for the Maven resolver”, behind Leiningen, naturally. As its name indicates, Leiningen emerged as a Maven wrapper. The indication is not so much in the name as in the reference to a short story, Leiningen versus the Ants, echoing the rivalry between Maven and Ant, tools that at one point were competing for mind-share in the Java world.

tl;dr By introducing richer semantics when talking about dependencies, tools.deps is enabling a new generation of Clojure tooling.

It’s interesting to note that Maven advertises itself as project management software, eschewing the terminology of build automation. Indeed, it hammers the point at every opportunity: our scope is broad. The notion of project is taken to include things such as documentation, version control, reporting, tests, etc. Project comprehension is how Maven sometimes refers to this. This openendedness is likely to have contributed to Maven’s resilience, its dominant position in the java landscape.

The Project Object Model, or POM in short, codifies a project along all its axes and dimensions. The ability to compile Clojure source code from a POM exists since February 2009, when Mark Derricutt pushed the first commit of the Clojure Maven plugin. That was seven months before Leiningen emerged. While the latter became the tool of choice in the Clojure community, the Clojure Maven plugin continued to exist in its own timeline.

What if Clojure projects were being built with mvn install and not lein install? Strangely enough, that alternate history became my reality. When tools.deps was released, I saw that it had the ability to generate POM files from a deps.edn. Those were bare bones and unusable as such, but if you tweaked them, and crucially, summoned the Clojure compiler at the right phase, you could build jars and uberjars straight out the box. What else could you get for free? It turned out: a whole lot.

The Maven model is predicated on a build lifecycle with different phases where goals can be attached and executed. To hook into that lifecycle, one writes mojos, a pun on pojos (Maven plain Old Java Object). One or several related mojos packaged together constitute a plugin, and plugins are to be found for nearly every thing that comes to mind due to Maven’s penetration and reach.

My strategy was simple but effective: muster desired Maven plugins and coax them in a custom POM that could then be executed via Maven. I named the project Meyvn. An astute reader might note that so far my efforts seemed to replicate those of Leiningen: yet another a Maven wrapper. I would concede them the point with the following caveat: Meyvn is a next-gen build tool. With that I mean a tool that incorporates the semantics of deps.edn.

The checkouts feature illustrates this point aptly. In Leiningen, you need to create a magic directory and symbolic links to extraneous source trees. Boot’s checkouts facility involves a configuration option and a command invocation in the directory of the projects whose source you want to pull. Today, thanks to tools.deps, we have a way to indicate that dependencies are on disk via the :local/root key. When you launch a REPL with Meyvn, it will recognize local dependencies from that notation, and as such will be able to track changes in the source trees and reload the project as necessary.

By taking a stance as to how dependencies should be defined, characterized and resolved, Cognitect’s tools.deps drew a line in the sand. None of the build tools would have been the same had tools.deps been around at their inception. And so, it is only natural that the release of tools.deps heralded a new age of Clojure tooling. Meyvn was a happy accident that occurred in my little bubble. Outside, a reconfiguration of practices was taking place. Some users were happy to ditch the all-in-one build tools of yore in favor of a tailored user-level deps.edn, striving to replace piecemeal the lost functionality. Others were sticking to the traditions, seeing no reason to upset workflows with a proven track record.

Today, Meyvn builds my artifacts and sets up the interactive environment in which work. It is still early days. As hinted by Alex Miller, tools.build is coming, and while we don’t have the details just yet, I wouldn’t be surprised if it further leverages the foundation laid by tools.deps. That of which I’m sure is that Maven will still be the invisible giant with raised shoulders.
